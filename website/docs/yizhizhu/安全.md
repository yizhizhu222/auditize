# 安全——扫描引擎 + 认证 + HTTPS

## 安全的三个层面

```
你的项目涉及的安全，分三层：

第一层：用户登录安全         ← auth.py（JWT + TOTP）
第二层：代码安全扫描         ← scanner/（这个最特别）
第三层：传输安全             ← HTTPS（Cloudflare 免费提供）
```

## 第一层：用户登录安全——JWT + TOTP

### JWT（JSON Web Token）

JWT 是一种让用户"拿着令牌通行"的机制。

```
用户登录 → 后端签发 JWT → 浏览器保存 JWT → 每次请求带上 JWT
```

**为什么不用 session：**
```
传统 session：用户登录 → 服务器记下"用户A已登录"
              → 用户每次请求带一个 session ID → 服务器查数据库验证
              → 问题：服务器要存 session，多台服务器时要共享

JWT：用户登录 → 服务器签一个加密的 token
     → 用户每次请求带 token → 服务器解密验证
     → 好处：服务器不需要存 session，token 本身就是证据
```

**JWT 的加密过程：**
```python
# 后端用 SECRET_KEY 加密 JWT
token = pyjwt.encode(
    {"user_id": 1, "username": "admin", "exp": 明天},
    SECRET_KEY,  # 配置文件里的密钥
    algorithm="HS256"
)

# 用户拿着 token 来请求时，后端解密验证
data = pyjwt.decode(token, SECRET_KEY, algorithms=["HS256"])
```

**为什么 SECRET_KEY 要保密？** 如果有人拿到了你的 SECRET_KEY，他可以伪造任意用户的 JWT，以 admin 身份登录你的系统。

### TOTP（双因子认证）

```
用户登录时：
  1. 输密码 → 后端验证密码对不对
  2. 输 6 位动态码 → 后端验证动态码对不对

动态码怎么生成：
  用户手机上（Google Authenticator）有一个密钥
  每 30 秒，密钥 + 当前时间 = 一个新的 6 位数
  后端也有同样的密钥，做同样的计算
  如果用户输入的数和后端算的数一样 → 通过
```

### CORS（跨域安全）

**什么是跨域？**
```
你的前端在 http://localhost:5173
你的后端在 http://localhost:8000
域名不同 → 浏览器认为这是"跨域请求"
默认情况下，浏览器会阻止前端向后端发请求
```

**CORS 怎么解决？**
```python
# 后端告诉浏览器："允许这些域名访问我"
origins = ["http://localhost:5173", "https://trufflekit.com"]
app.add_middleware(CORSMiddleware, allow_origins=origins)
```

你的 CORS 配置：
```python
origins = "http://localhost:5173,http://127.0.0.1:5173,http://58.16.61.185,..."
```

这个配置说："这些域名发来的请求，我都接受。"

## 第二层：安全扫描引擎（你最特别的功能）

### 为什么需要安全扫描

```
AI 生成的代码有一个问题：
  AI 不知道你的使用场景
  AI 可能会生成不安全的代码
  而你不懂代码，看不出来哪里不安全
```

安全扫描引擎帮你看：
```
AI 生成的代码 → 扫描引擎 → "这段代码第 3 行有一个高危问题"
```

### 检测方式

安全扫描引擎使用两种检测方式：**AST 分析 + 正则匹配**

#### AST 分析（Abstract Syntax Tree）

AST = 把代码解析成树状结构。

```python
# 用户写的代码：
code = """
import os
os.system("rm -rf /")
"""

# 解析成树：
Module
├── Import("os")
└── Call
    ├── Attribute(Name("os"), "system")
    └── Constant("rm -rf /")

# 遍历这棵树，找到所有函数调用
# 发现 os.system() → 危险！
```

AST 的好处——只分析代码结构，不运行代码，安全且快速。

#### 正则匹配（Regular Expression）

正则 = 在文本中找"模式"。

```
找 API Key:
  api_key = "sk-123456"      ✓ 匹配
  APIKEY=sk-123456           ✓ 匹配
  my_secret = "hello"        ✗ 不匹配（没有 api_key 字样）

找密码:
  password = "123456"        ✓ 匹配
  passwd: 123456             ✓ 匹配
  password_hash = "abc"      ✓ 匹配（虽然这个是加密后的）
```

### 你的扫描器能检测什么

| 类别 | 严重程度 | 检测方式 | 例子 |
|------|---------|---------|------|
| **危险函数** | Critical/High | AST | `eval()`、`exec()`、`os.system()` |
| **命令注入** | High | AST | `subprocess.run()`、`popen()` |
| **SQL 注入** | High | 正则 | `cursor.execute(f"SELECT * FROM {user_input}")` |
| **API Key 泄露** | High | 正则 | `api_key = "sk-xxx"` |
| **密码硬编码** | High | 正则 | `password = "123456"` |
| **私钥泄露** | Critical | 正则 | `-----BEGIN PRIVATE KEY-----` |
| **网络请求** | Medium | 正则 | `requests.get()`、`fetch()` |
| **文件操作** | Medium | 正则 | `open()`、`os.remove()` |
| **XSS 风险** | Medium | 正则 | `innerHTML = data`（JS） |
| **缓冲区溢出** | High | 正则 | `strcpy()`（C++） |

### 评分系统

```python
# 每条发现有自己的权重
RISK_WEIGHTS = {
    "critical": 40,   # 一个 critical = 40 分
    "high": 20,       # 一个 high = 20 分
    "medium": 10,
    "low": 5,
    "info": 0
}

# 总分 = 所有发现的权重加起来（最高 100 分）
# 0 分 = 安全
# 1-20 分 = 轻微问题
# 30-59 分 = 建议审查
# 60+ 分 = 危险
```

### 支持的语言

| 语言 | 检测方式 | 检测项数量 |
|------|---------|-----------|
| Python | AST + 正则 | 最多（20+ 个检测项） |
| JavaScript | 正则 | 10 个检测项 |
| Go | 正则 | 6 个检测项 |
| C++ | 正则 | 10 个检测项 |

### 为什么不支持其他语言？

扫描器是逐行写检测规则的，每种语言有不同的危险函数和语法。
要支持新语言就要写新的规则。目前这 4 种语言覆盖了最常见的项目类型。

### 报告示例

```
安全扫描报告：

评分: 40/100 ｜ 结论: 🟠 Needs Review

What this code does:
  • 运行系统命令（os.system）
  • 包含硬编码密码

发现的问题:
  1. [高危] 第 3 行: os.system() — 运行系统命令
     代码: os.system("rm -rf /")
     建议: 避免使用 os.system，改用 subprocess 并验证输入

  2. [高危] 第 5 行: 硬编码密码
     代码: password = "123456"
     建议: 把密码放在环境变量或配置文件中
```

## 第三层：传输安全——HTTPS

### 什么是 HTTPS？

```
HTTP：   你访问网站，数据明文传输
        中间人可以拦截看到你的密码

HTTPS：  你访问网站，数据加密传输
        中间人拦截了也看不懂
```

### 你的 HTTPS 是怎么来的

```
你用了 Cloudflare Tunnel → Cloudflare 自动给你 HTTPS 证书
                              不需要自己申请、不需要花钱
                              浏览器看到证书 → "这个网站是安全的"
```

## 其他安全问题

### 密码存储——不是明文存的

```python
# 注册时：
salt = secrets.token_hex(8)  # 随机 8 字节
password_hash = sha256(salt + password)  # 加密
# 数据库存的是：salt 和 password_hash
# 原始密码不存

# 登录时：
# 取出 salt 和 password_hash
# 计算 sha256(salt + 用户输入的密码)
# 比较计算结果和数据库里的 password_hash
```

### 你以前有一个安全漏洞——"000000"万能码

```python
# 问题（已修复）：
if req.token == "000000":
    # 万能码：不验证密码，直接以第一个用户身份登录
    # 任何知道这个"技巧"的人都能以 admin 登录

# 修复：删掉了这个判断
```

### 常见安全建议

| 建议 | 说明 |
|------|------|
| SECRET_KEY 不要提交到 GitHub | 有人拿到你的密钥就能伪造管理员身份 |
| 不要相信用户输入 | 用户发的数据可能包含恶意代码 |
| 使用 HTTPS | 没有 HTTPS 的数据可以被任何人看到 |

## 学习要点总结

### 你从安全这部分学到了什么

1. **JWT 认证机制**——token 替代 session，服务器不需要存登录状态
2. **TOTP 双因子**——密码 + 动态码，两层保护
3. **AST 代码分析**——把代码解析成树结构遍历，不运行代码检查安全问题
4. **正则匹配**——在文本中找敏感模式（API Key、密码等）
5. **CORS 安全策略**——不让随便哪个网站都能访问你的 API
6. **HTTPS 的重要性**——不加密的数据等于明文传输
7. **密码不要明文存**——加盐哈希，就算数据库泄露也拿不到密码

## 产品方向：Truffle Security 引擎化 + CLI + 私有化

### 要解决的问题

1. **安全扫描规则与代码耦合** — 每次加检测项都要改 Python 代码，无法热更新
2. **无法区分 SaaS/私有化/CLI** — 一套代码需要同时覆盖三种交付形态
3. **AI 生成代码的安全检测是个新赛道** — 传统安全工具（Snyk/奇安信）面向开发者，你的差异化是面向非技术人员也能看懂的报告
4. **大企业不会买，但 99% 的中小企业没有安全团队** — 目标客户是 50-500 人、用 AI 写代码但没有安全团队的公司

### 主要方法

1. **规则与引擎分离** — 检测规则从 Python 硬编码改为外部 JSON 文件放到 `rules/` 目录，引擎从目录加载，加新规则 = 放一个新 JSON 文件
2. **QQ 式规则更新** — 企业版启动时检查云端规则版本，下载新规则包 → 解压覆盖 → `reload()` 热加载，无需升级整个软件
3. **三形态交付**：
   - CLI：`truffle scan` 终端运行，代码不出用户机器，零服务器成本
   - API：开发者集成到 CI/CD，`curl` 或 GitHub Actions 调用
   - Docker 私有化部署：`docker compose up -d` 一键启动，给不能出内网的企业
4. **官网免费扫描引流**：打开网页就能用，不需要注册，靠免费拉量建立品牌认知
5. **不跟传统安全产品正面竞争**，切 "AI Generated Code Security" 新赛道，做最早入局的人
6. **技术壁垒不在代码，在规则库的持续维护** — 每周追新漏洞、降误报、覆盖新语言，这是纯人工投入的护城河
