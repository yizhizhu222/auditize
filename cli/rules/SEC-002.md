# SEC-002: .env file not protected by .gitignore

| 字段 | 值 |
|---|---|
| 规则编号 | SEC-002 |
| 分类 | 安全 (Security) |
| 严重等级 | 🔴 Critical |

## 为什么这是问题

`.env` 文件通常包含数据库密码、API 密钥、JWT 密钥、云服务凭证等敏感信息。如果 `.env` 文件未被添加到 `.gitignore` 中，它会被 Git 跟踪并推送到远程仓库。对于公开仓库，这意味着所有凭证立即暴露给全世界。即使是私有仓库，凭证也会暴露给所有拥有仓库访问权限的团队成员和 CI/CD 系统，增加了泄露面。

一旦 `.env` 文件被提交，即使后续将其加入 `.gitignore`，它仍然存在于 Git 历史中。自动化扫描工具和攻击者会专门搜索此类文件。

## 修复方法

1. **立即检查 `.env` 是否已被提交**：运行 `git status` 或 `git ls-files .env`。
2. **如果已提交**：立即轮换所有在 `.env` 中的凭证。
3. **将 `.env` 加入 `.gitignore`**：创建或编辑 `.gitignore` 文件，添加 `.env` 条目。
4. **创建 `.env.example` 文件**：提交一个不含真实值的示例文件，列出所需的环境变量名称。
5. **如果历史中已有记录**：使用 `git filter-branch` 或 BFG Repo-Cleaner 从 Git 历史中移除文件。

## 示例

```
# ❌ 错误的做法：.env 被 Git 跟踪
# .gitignore 文件中缺少 .env 条目

# .gitignore (不完整)
node_modules/
dist/
```

```
# ✅ 正确的做法
# .gitignore 中包含 .env
node_modules/
dist/
.env
.env.local
.env.production
```

```
# .env.example (可安全提交的参考文件)
DATABASE_URL=postgresql://localhost:5432/mydb
API_KEY=your_api_key_here
SECRET_KEY=change_this_to_a_random_secret
```
