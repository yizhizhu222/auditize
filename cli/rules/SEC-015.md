# SEC-015: Empty catch/except blocks

| 字段 | 值 |
|---|---|
| 规则编号 | SEC-015 |
| 分类 | 代码质量 (Code Quality) |
| 严重等级 | 🟠 High |

## 为什么这是问题

空的 `catch` 或 `except` 块会静默吞掉所有异常，导致程序在错误状态下继续运行而不留下任何日志。这使得调试变得极其困难——开发者和运维人员无法知道错误发生了、发生在哪里、以及错误的具体原因是什么。在生产环境中，这可能导致数据损坏、支付失败、资源泄露等严重问题而不被察觉。

更危险的是，攻击者可能利用空 catch 块隐藏的攻击路径。例如，如果认证函数的异常被静默吞掉，可能导致认证检查绕过。空 catch 块通常源于"以后再处理"的侥幸心态，但往往永远不会被补上。

## 修复方法

1. **绝不使用空的 catch/except 块**：至少记录错误日志。
2. **缩小异常捕获范围**：只捕获你知道如何处理的特定异常类型。
3. **至少记录日志**：在 catch 块中添加有意义的错误日志。
4. **考虑是否应该向上传播**：如果你无法在此处处理异常，让它向上传递。
5. **配置 Linter 规则**：启用 `no-empty`（ESLint）、`no-explicit-any`（TypeScript）等规则。

## 示例

```javascript
// ❌ 错误的做法
try {
    const result = await processPayment(userId, amount);
} catch (error) {
    // 什么都不做 —— 错误被静默吞掉！
}

// ❌ 同样危险 —— 捕获所有异常但不处理
try {
    const data = JSON.parse(rawData);
    saveToDatabase(data);
} catch (_) {}
```

```javascript
// ✅ 正确的做法
try {
    const result = await processPayment(userId, amount);
} catch (error) {
    // 至少记录日志
    logger.error("Payment processing failed", {
        userId,
        amount,
        error: error.message,
        stack: error.stack,
    });
    // 向上传播或处理
    throw new PaymentError("Payment processing failed", { cause: error });
}
```

```python
# ❌ 错误的做法
try:
    user = db.query(User).filter(id=user_id).one()
except:
    pass  # 静默忽略数据库错误，返回 None 可能导致后续空指针

# ✅ 正确的做法
try:
    user = db.query(User).filter(id=user_id).one()
except UserNotFoundError:
    logger.warning(f"User {user_id} not found")
    return None
except DatabaseError as e:
    logger.error(f"Database error fetching user {user_id}: {e}")
    raise
```
