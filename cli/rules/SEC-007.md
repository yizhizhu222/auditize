# SEC-007: TODO/FIXME accumulation

| 字段 | 值 |
|---|---|
| 规则编号 | SEC-007 |
| 分类 | 代码质量 (Code Quality) |
| 严重等级 | 🔵 Low |

## 为什么这是问题

大量未处理的 `TODO` 和 `FIXME` 注释表明代码中存在已知的未解决问题、技术债务或潜在缺陷。`TODO` 通常标记未完成的功能实现，而 `FIXME` 标记已知的 bug 或问题。如果这些标记被长期忽视，它们所指向的问题可能在生产环境中表现为 bug、性能问题或安全漏洞。

在 CI/CD 审查中，大量累积的 TODO/FIXME 使代码审查变得困难，新开发者也难以判断哪些标记仍相关、哪些已经过时。此外，`FIXME` 标记的安全相关问题尤其危险——它们代表的已知漏洞可能被攻击者利用。

## 修复方法

1. **审查所有 TODO/FIXME 标记**：评估每个标记是否仍有必要处理。
2. **处理 FIXME 优先**：FIXME 通常代表已发现的 bug，应尽快修复。
3. **为关键 TODO 创建 Issues**：将重要的未完成任务从注释迁移到项目管理工具（如 GitHub Issues、Jira）。
4. **移除已过时的注释**：已完成或不再相关的标记应直接删除。
5. **在 CI 中设置限制**：可以使用工具（如 `eslint-plugin-no-todo`）禁止在 main 分支中添加新的 TODO/FIXME，或限制其数量。
6. **添加日期和责任人**：如果必须留下 TODO，附上创建日期和责任人信息。

## 示例

```python
# ❌ 错误的做法：无信息、长期存在的 TODO
def process_user_data(data):
    # TODO: 处理边界情况
    # FIXME: 这里有问题
    # TODO: 以后优化
    # TODO: 添加错误处理
    return data["name"]

# ✅ 正确的做法：可追踪的 TODO，定期清理
def process_user_data(data):
    # TODO(zhangsan, 2024-01-15): 处理空值和 None 的边界情况
    # 相关 Issue: #123
    if data is None:
        return None
    return data.get("name", "")

# ✅ 更好的做法：直接在项目管理工具中跟踪
# 而不是在代码中遗留 TODO 注释
```
