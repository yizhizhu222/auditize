# SEC-012: Docker ports exposed to public

| 字段 | 值 |
|---|---|
| 规则编号 | SEC-012 |
| 分类 | 安全 (Security) |
| 严重等级 | 🟠 High |

## 为什么这是问题

在 Docker Compose 或 Dockerfile 中将容器端口绑定到 `0.0.0.0` 或直接暴露到外网（如 `"0.0.0.0:3306:3306"`、`"5432:5432"`）会使内部服务（数据库、缓存、管理面板等）对公网开放。这些服务通常没有配置足够的安全防护（如 IP 白名单、身份验证），容易被扫描工具发现并成为攻击目标。

数据库端口（MySQL: 3306、PostgreSQL: 5432、Redis: 6379、MongoDB: 27017）暴露到公网是极为常见的错误。攻击者使用 Shodan、ZoomEye 等搜索引擎可以快速发现这些服务，并尝试默认密码或已知漏洞进行攻击。

## 修复方法

1. **仅暴露必要的端口**：只将需要外部访问的服务端口绑定到 `0.0.0.0`（如 Web 服务的 80/443）。
2. **使用 internal 网络**：数据库、缓存等内部服务应仅连接到 Docker 内部网络，不对外暴露端口。
3. **绑定到 localhost**：如果需要在宿主机访问内部服务，绑定到 `127.0.0.1`：`"127.0.0.1:3306:3306"`。
4. **配置防火墙**：在云服务商的安全组规则中限制入站流量。
5. **使用反向代理**：通过 Nginx/Traefik 统一管理对外暴露的端口，内部服务不直接暴露。

## 示例

```yaml
# ❌ 错误的做法：数据库端口直接暴露到公网
services:
  web:
    image: myapp
    ports:
      - "80:80"
      - "443:443"

  db:
    image: mysql:8
    ports:
      - "3306:3306"    # ← 数据库对公网开放！
    environment:
      MYSQL_ROOT_PASSWORD: secret

  redis:
    image: redis:7
    ports:
      - "0.0.0.0:6379:6379"  # ← Redis 无认证暴露到公网！
```

```yaml
# ✅ 正确的做法
services:
  web:
    image: myapp
    ports:
      - "80:80"         # Web 服务需要对外暴露
      - "443:443"
    networks:
      - frontend
      - backend

  db:
    image: mysql:8
    # 不暴露任何端口到宿主机，仅在内部网络可访问
    expose:
      - "3306"
    networks:
      - backend
    environment:
      MYSQL_ROOT_PASSWORD: ${DB_PASSWORD}

  redis:
    image: redis:7
    expose:
      - "6379"
    networks:
      - backend

  # 可选：仅本地开发时访问数据库
  adminer:
    image: adminer
    ports:
      - "127.0.0.1:8080:8080"  # 仅绑定到本地
    networks:
      - backend

networks:
  frontend:
  backend:
    internal: true  # 后端网络不对外
```
