# SEC-011: Large files in repo

| 字段 | 值 |
|---|---|
| 规则编号 | SEC-011 |
| 分类 | 性能 (Performance) |
| 严重等级 | 🟡 Medium |

## 为什么这是问题

Git 并非为大文件管理而设计。将超过 50MB 的大文件（如二进制文件、视频、音频、数据集、编译产物、打包后的依赖）提交到 Git 仓库会导致一系列问题：仓库克隆时间暴涨、`git clone` 和 `git pull` 操作变慢、磁盘使用量膨胀、`git log` 等操作变迟钝。一旦大文件被提交，即使后续删除，它仍然存在于 Git 对象库中，每次克隆都会下载。

大文件还会影响团队协作效率——每个开发者都需要下载这些他们可能根本不需要的文件。对于 CI/CD 流水线，大文件会显著增加构建时间。常见的例子包括：`node_modules.zip`、`dataset.csv`（几百 MB）、`model.pt`（机器学习模型文件）。

## 修复方法

1. **识别大文件**：使用 `git ls-tree -r -t -l --full-name HEAD | sort -n -k 4 | tail -10` 查看最大的文件。
2. **将大文件从仓库中移除**：使用 BFG Repo-Cleaner 或 `git filter-branch` 从历史中清理。
3. **使用 Git LFS**：对于需要版本控制的二进制文件（如设计资源、模型文件），配置 Git Large File Storage。
4. **使用对象存储**：对于数据集和媒体文件，使用 AWS S3、Google Cloud Storage 等外部存储。
5. **在 `.gitignore` 中添加大文件模式**：如 `*.zip`、`*.tar.gz`、`*.mp4`、`*.pkl`。

## 示例

```bash
# ❌ 错误的做法：提交大文件到仓库
git add data/dataset.csv    # 500MB CSV 文件
git add models/weights.pt   # 2GB 模型文件
git add node_modules/       # 数十万小文件
git commit -m "Add data and models"

# ✅ 正确的做法：使用 Git LFS 或外部存储
# 方案1：Git LFS
git lfs track "*.csv"
git lfs track "*.pt"
git lfs track "*.zip"
git add .gitattributes
git add data/dataset.csv  # 实际存储在 LFS 中
git commit -m "Add dataset via LFS"

# 方案2：使用外部存储并在代码中引用
DATA_URL = "https://s3.amazonaws.com/bucket/dataset.csv"
download_data(DATA_URL, local_path="data/dataset.csv")

# 识别仓库中大文件
find . -type f -size +10M | grep -v .git
# 输出示例:
# ./data/backup.sql (120MB)
# ./assets/video/demo.mp4 (250MB)
# ./model.pkl (800MB)
```

```
# .gitattributes (Git LFS 配置)
*.csv filter=lfs diff=lfs merge=lfs -text
*.pt filter=lfs diff=lfs merge=lfs -text
*.pkl filter=lfs diff=lfs merge=lfs -text
*.mp4 filter=lfs diff=lfs merge=lfs -text
```
