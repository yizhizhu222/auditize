# SEC-019: vendor/node_modules committed to git

| 字段 | 值 |
|---|---|
| 规则编号 | SEC-019 |
| 分类 | 安全 (Security) |
| 严重等级 | 🟠 High |

## 为什么这是问题

将 `vendor/`、`node_modules/`、`packages/` 等依赖目录提交到 Git 仓库会导致多个严重问题。首先，这些目录包含成千上万的小文件，会严重降低 Git 操作性能——`git clone`、`git status`、`git diff` 等操作将异常缓慢。一个典型的 Node.js 项目可能有 5 万到 10 万个文件在这一目录中。

其次，安全风险巨大——这些第三方依赖中可能包含已知的 CVE 漏洞，且当依赖更新修复漏洞时，如果你提交了 `node_modules/`，很难追踪到你使用的是哪个版本的代码。依赖目录被提交后，`package-lock.json` 或 `yarn.lock` 的锁定作用就失去了意义，因为实际代码已经存在于仓库中，覆盖了 lock 文件的锁定机制。

## 修复方法

1. **立即将 `node_modules/`、`vendor/` 等加入 `.gitignore`**。
2. **使用 lock 文件代替**：提交 `package-lock.json`、`yarn.lock`、`composer.lock` 等来锁定依赖版本。
3. **在 CI/CD 中运行 `npm ci`**（或对应工具）来安装精确版本的依赖。
4. **如果已经提交**：从 Git 追踪中移除并清理历史。
5. **对于需要离线安装的场景**：使用私有 npm registry 或 Artifactory 缓存。

## 示例

```
# ❌ 错误的做法：node_modules 被 git 追踪
# git ls-files | head -5 显示
node_modules/express/lib/...
node_modules/lodash/lodash.js
node_modules/axios/index.js
...

# .gitignore 中缺少 node_modules/
# 仓库体积巨大，克隆缓慢

# ✅ 正确的做法
# .gitignore
node_modules/
vendor/
.pnp
.pnp.js
__pycache__/
*.egg-info/

# 提交 lock 文件来锁定依赖
git add package.json package-lock.json
git add yarn.lock

# 在 CI/CD 中使用精确安装
# package.json 锁定版本 + package-lock.json
npm ci --only=production

# 如果已经提交，从追踪中移除
git rm -r --cached node_modules/
git rm -r --cached vendor/
git commit -m "Remove committed dependencies"
```

```dockerfile
# ✅ Docker 中的正确做法：不在构建中安装生产依赖
FROM node:20-alpine

WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci --only=production  # 从 registry 安装，不复制本地 node_modules
COPY . .
```
