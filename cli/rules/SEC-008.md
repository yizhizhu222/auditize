# SEC-008: Sensitive file permissions

| 字段 | 值 |
|---|---|
| 规则编号 | SEC-008 |
| 分类 | 安全 (Security) |
| 严重等级 | 🟡 Medium |

## 为什么这是问题

敏感文件（如密钥、配置文件、数据库文件）拥有过于宽松的权限设置（如 `777`、`755`、`644`）可能导致未经授权的用户、进程或容器读取或修改这些文件。在多用户系统、共享托管环境或容器化部署中，不正确的文件权限可能使攻击者在获得有限的系统访问权限后，读取数据库密码、私钥或其他敏感配置。

常见的风险场景包括：配置文件被设为全局可读（`644`）导致其他系统用户可读取数据库密码；脚本文件被设为全局可写（`777`）导致攻击者可注入恶意代码；`.git` 目录权限不正确导致攻击者获取版本历史信息。

## 修复方法

1. **审查所有敏感文件的权限**：重点关注配置文件、密钥文件、数据库文件和 `.git` 目录。
2. **设置最小必要权限**：
   - 配置文件（含密码）：`600`（仅所有者可读写）或 `640`（所有者读写、组只读）
   - 私钥文件：`400` 或 `600`
   - 可执行脚本：`700` 或 `750`
   - `.git` 目录：`700`
3. **添加权限检查到 CI/CD**：在部署流程中检查关键目录和文件的权限设置。
4. **使用 `chmod` 修复权限**：在部署脚本中自动设置正确的权限。

## 示例

```bash
# ❌ 错误的做法：文件权限过于宽松
# 敏感文件被设置为全局可读/写
config/database.yml: -rwxrwxrwx (777)
secrets/private.key: -rwxr-xr-x (755)
.env: -rw-rw-rw- (666)

# ✅ 正确的做法：最小权限原则
# 设置配置文件权限
chmod 600 config/database.yml
# 设置私钥权限
chmod 400 secrets/private.key
# 设置 .env 文件权限
chmod 600 .env
# 设置 .git 目录权限
chmod -R 700 .git/
# 设置脚本权限
chmod 750 scripts/deploy.sh

# 验证权限的命令
find . -name "*.pem" -o -name "*.key" -o -name ".env" | xargs ls -la
```

```
# ❌ 常见的危险权限配置
AWS_KEY_PATH=/home/app/keys
chmod 777 -R /home/app/keys/

# ✅ 安全的权限配置
AWS_KEY_PATH=/home/app/keys
chmod 700 $AWS_KEY_PATH
chmod 400 $AWS_KEY_PATH/aws-key.pem
```
