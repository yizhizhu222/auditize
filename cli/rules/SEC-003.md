# SEC-003: Git history contains sensitive files

| 字段 | 值 |
|---|---|
| 规则编号 | SEC-003 |
| 分类 | 安全 (Security) |
| 严重等级 | 🔴 Critical |

## 为什么这是问题

Git 历史中即使只存在一次敏感文件（如包含密码的配置文件、私钥、凭据文件），也会造成永久性安全风险。任何能够访问仓库的人都可以通过 `git log`、`git show` 或浏览历史来恢复这些文件。即使你在后续提交中删除了文件，它仍然存在于 Git 对象数据库中，可以通过 `git checkout` 到历史提交来访问。

公共仓库尤其危险——自动化爬虫会持续扫描公开仓库的完整 Git 历史，搜索常见的凭据模式和文件名称。私有仓库同样存在风险，因为所有协作者都有权访问完整历史。

## 修复方法

1. **立即轮换所有泄露的凭证**：在任何历史记录被清理之前，先确保凭证已失效。
2. **使用 BFG Repo-Cleaner**：`bfg --delete-files config-with-keys.json` 是最简单的方法。
3. **或使用 git filter-branch**：`git filter-branch --force --index-filter 'git rm --cached --ignore-unmatch secrets.txt' --prune-empty --tag-name-filter cat -- --all`
4. **强制推送重写后的历史**：`git push origin --force --all`（需要与团队协调）。
5. **通知所有协作者**：他们需要重新克隆仓库，因为旧历史已被重写。

## 示例

```bash
# ❌ 错误的做法：敏感文件存在于 git 历史中
git add config/database.yml  # 包含真实密码
git commit -m "Add config"
# 后来删除了密码，但没有清理历史

# ✅ 正确的做法：从历史中彻底清除
# 方法1：使用 BFG Repo-Cleaner
bfg --delete-files database.yml --delete-folders secrets .

# 方法2：使用 git filter-branch
git filter-branch --force --index-filter \
  "git rm --cached --ignore-unmatch '*.pem' 'secrets*'" \
  --prune-empty --tag-name-filter cat -- --all

# 完成后的清理
git reflog expire --expire=now --all
git gc --aggressive --prune=now
git push origin --force --all
```
