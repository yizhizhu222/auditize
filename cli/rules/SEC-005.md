# SEC-005: SSL private keys in repo

| 字段 | 值 |
|---|---|
| 规则编号 | SEC-005 |
| 分类 | 安全 (Security) |
| 严重等级 | 🔴 Critical |

## 为什么这是问题

SSL/TLS 私钥（通常为 `.pem`、`.key`、`.cert` 文件）是实现 HTTPS 加密通信和身份验证的核心安全组件。如果私钥被提交到代码仓库，任何能够访问仓库的人都可以解密该服务器的加密通信流量、伪装成该服务器进行中间人攻击，或撤销/滥用该证书的信任关系。

私钥泄露的后果极为严重：你需要立刻吊销对应证书并重新签发，这会中断生产服务的 HTTPS 连接，并可能影响相关的 CDN、负载均衡器和客户端缓存。此外，如果私钥被用于服务间认证（如 mTLS），所有相关服务的通信都可能被劫持。

## 修复方法

1. **立即吊销证书**：联系证书颁发机构（CA）吊销泄露的证书。
2. **签发新证书**：使用新生成的私钥重新申请证书。
3. **从 Git 历史中移除私钥文件**：使用 BFG Repo-Cleaner 或 `git filter-branch`。
4. **私钥应存储在专用位置**：如 HashiCorp Vault、AWS Secrets Manager 或服务器上的安全目录（`/etc/ssl/private/`）。
5. **在 `.gitignore` 中添加 `*.pem`、`*.key`、`*.cert` 模式**。

## 示例

```bash
# ❌ 错误的做法：将私钥提交到仓库
# 在代码仓库中发现了这些文件：
#   server.key
#   ssl/certificate.pem
#   private/domain.key

# git log --oneline
# a1b2c3d Add SSL certificates for production

# ✅ 正确的做法：私钥永远不应进入仓库
# 在 .gitignore 中添加
*.key
*.pem
*.cert
*.p12
*.pfx

# 从 Git 追踪中移除（如果已提交）
git rm --cached server.key
echo "server.key" >> .gitignore

# 使用环境变量或密钥管理服务加载
# docker-compose.yml 示例
services:
  web:
    image: myapp
    volumes:
      - /etc/ssl/certs/server.pem:/etc/ssl/certs/server.pem:ro
      - /etc/ssl/private/server.key:/etc/ssl/private/server.key:ro
```
