# SEC-010: Database file exposed

| 字段 | 值 |
|---|---|
| 规则编号 | SEC-010 |
| 分类 | 安全 (Security) |
| 严重等级 | 🔴 Critical |

## 为什么这是问题

数据库文件（如 SQLite 的 `.db`、`.sqlite`、`.sqlite3` 文件）被提交到代码仓库中意味着所有数据可能被泄露。开发数据库中通常包含真实或接近真实的测试数据，可能包括用户账户、邮箱地址、密码哈希、支付记录等敏感信息。即使是空的数据库模式文件，也可能暴露数据库结构和表关系，便于攻击者进行 SQL 注入或其它攻击。

此外，SQLite 数据库文件是二进制文件，Git 无法有效进行差异比较，会随着每次变更增加仓库体积。如果数据库文件在开发过程中频繁变化（如每次运行测试都写入数据），仓库会快速膨胀。

## 修复方法

1. **立即将数据库文件加入 `.gitignore`**：添加 `*.db`、`*.sqlite`、`*.sqlite3` 等模式。
2. **如果已经提交**：使用 BFG Repo-Cleaner 从历史中移除。
3. **提供数据库模式文件**：提交一个 SQL 脚本或迁移文件来重建数据库结构。
4. **使用数据库迁移工具**：如 Alembic、Prisma Migrate、Knex.js 来管理数据库版本。
5. **使用 seed 脚本**：提供用于生成测试数据的脚本，而不是提交数据库文件本身。

## 示例

```
# ❌ 错误的做法：数据库文件被 Git 跟踪
# git ls-files 显示：
#   data/development.db
#   data/production.sqlite3
#   test/test_data.db

# ✅ 正确的做法：在 .gitignore 中添加
*.db
*.sqlite
*.sqlite3
*.dblite
data/

# 提交数据库迁移脚本和 seed 脚本
db/
  migrations/
    001_create_users.sql
    002_create_orders.sql
  seeds/
    seed_data.py
schema.sql

# 使用迁移工具创建数据库
migrate-db --url $DATABASE_URL db/migrations
seed-db --url $DATABASE_URL db/seeds
```

```python
# ❌ 错误的做法：在代码中直接引用本地数据库文件
conn = sqlite3.connect("data/development.db")

# ✅ 正确的做法：使用环境变量配置数据库路径
import os
DATABASE_URL = os.getenv(
    "DATABASE_URL",
    "sqlite:///./dev.db"  # 仅在本地开发时使用默认值
)
```
