# SEC-013: Docker latest tag

| 字段 | 值 |
|---|---|
| 规则编号 | SEC-013 |
| 分类 | 安全 (Security) |
| 严重等级 | 🟡 Medium |

## 为什么这是问题

在 Docker 配置中使用 `latest` 标签（或在 `docker-compose.yml` 中不指定标签，默认使用 `latest`）会导致不可复现的构建和部署。`latest` 标签是一个会随时间变化的可变引用——今天构建成功的镜像，下周再构建可能使用完全不同版本的 base image，其中可能包含破坏性的 API 变更、引入新的 bug，甚至包含安全漏洞。

在生产环境中，不可复现的部署是灾难性的。如果 `latest` 标签指向了一个有问题的版本，回滚变得困难，因为你不知道之前使用的是哪个具体版本。此外，攻击者也可能通过替换或篡改 `latest` 标签来进行供应链攻击。

## 修复方法

1. **始终使用精确版本标签**：如 `node:20.11.0-alpine` 而不是 `node:latest`。
2. **在 CI/CD 中生成唯一标签**：使用 Git commit SHA、语义版本号或构建时间戳作为镜像标签。
3. **使用固定版本的依赖镜像**：为所有基础镜像和中间件镜像指定明确的版本号。
4. **定期更新并测试**：定期手动更新到新版本，并在测试环境中充分验证后再部署到生产。
5. **使用 Dependabot/Renovate 自动管理**：配置自动依赖更新工具来跟踪 base image 的版本变更。

## 示例

```yaml
# ❌ 错误的做法：使用 latest 标签
# docker-compose.yml
services:
  web:
    image: node:latest     # ← 每次部署可能不同版本
    build: .

  db:
    image: mysql:latest    # ← 不可复现的数据库镜像
    environment:
      MYSQL_ROOT_PASSWORD: secret

  cache:
    image: redis           # ← 不指定标签 = latest
```

```yaml
# ✅ 正确的做法：使用精确版本标签
# docker-compose.yml
services:
  web:
    build:
      context: .
      dockerfile: Dockerfile
    image: myapp:${BUILD_TAG:-latest}  # CI 中传入具体版本

  db:
    image: mysql:8.0.35   # 固定的次要版本
    environment:
      MYSQL_ROOT_PASSWORD: ${DB_PASSWORD}

  cache:
    image: redis:7.2.4-alpine  # 固定版本 + 轻量基础镜像
```

```dockerfile
# ❌ 错误的做法：Dockerfile 使用 latest
FROM node:latest
RUN npm install

# ✅ 正确的做法：Dockerfile 使用精确版本
FROM node:20.11.0-alpine@sha256:abc123def456
RUN npm ci --only=production
```
