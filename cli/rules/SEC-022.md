# SEC-022: DEBUG mode enabled in production

| 字段 | 值 |
|---|---|
| 规则编号 | SEC-022 |
| 分类 | 安全 (Security) |
| 严重等级 | 🔴 Critical |

## 为什么这是问题

在生产环境中启用 DEBUG 模式（如 Flask 的 `debug=True`、Django 的 `DEBUG=True`、Express 的 `NODE_ENV=development`）会暴露大量敏感信息。在 DEBUG 模式下，应用错误时会显示详细的堆栈跟踪、源代码片段、环境变量和配置信息。这些信息对攻击者极为有价值，可以用来了解应用的内部架构、发现漏洞点、甚至直接获取数据库密码和 API 密钥。

此外，某些框架的 DEBUG 模式还会提供交互式调试器（如 Werkzeug 的调试器），允许执行任意代码。攻击者一旦触发一个错误，就可以在服务器上执行任何 Python/JavaScript 代码，获得完整的远程代码执行（RCE）权限。这是一个极其严重的安全漏洞。

## 修复方法

1. **使用环境变量控制 DEBUG 模式**：确保生产环境的配置不启用 DEBUG。
2. **配置自定义错误页面**：在生产环境中使用通用的、不泄露信息的错误页面。
3. **集中式日志记录**：将错误详细信息发送到日志系统，而不是展示给用户。
4. **在 CI/CD 中检查**：确保部署到生产的代码中 `DEBUG` 被设置为 `False`。
5. **使用环境特定的配置文件**：为开发、测试、生产分别维护不同的配置文件。

## 示例

```python
# ❌ 错误的做法
# app.py
from flask import Flask

app = Flask(__name__)
app.config["DEBUG"] = True  # 如果部署到生产，危险！

# 或者
app.run(debug=True, host="0.0.0.0", port=5000)
```

```python
# ✅ 正确的做法
import os
from flask import Flask

app = Flask(__name__)

# 通过环境变量控制，生产环境不设置或设置为 False
app.config["DEBUG"] = os.getenv("FLASK_DEBUG", "False").lower() == "true"

# 或者使用生产/开发分离的配置类
class Config:
    SECRET_KEY = os.getenv("SECRET_KEY")
    DEBUG = False

class DevelopmentConfig(Config):
    DEBUG = True

# 根据环境选择配置
env = os.getenv("APP_ENV", "production")
if env == "development":
    app.config.from_object(DevelopmentConfig)
else:
    app.config.from_object(Config)
```

```javascript
// ❌ 错误的做法：Express
const express = require("express");
const app = express();

app.set("env", "development");  // 生产环境应使用 "production"

// ✅ 正确的做法
const app = express();
const isProduction = process.env.NODE_ENV === "production";

app.set("env", isProduction ? "production" : "development");

// 生产环境使用自定义错误处理
if (isProduction) {
    app.use((err, req, res, next) => {
        logger.error("Unhandled error", {
            error: err.message,
            stack: err.stack,
            path: req.path,
            method: req.method,
        });
        res.status(500).json({ error: "Internal server error" });
    });
}
```
